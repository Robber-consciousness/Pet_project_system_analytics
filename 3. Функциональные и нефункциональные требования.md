# 3. Функциональные и нефункциональные требования
# 3.1 Функциональные требования

## **Use Case 1**

**Сценарий:** Регистрация технологов/инженеров в системе. 

**Цель:** Успешно зарегистрироваться в SCADA для доступа к функционалу системы. 

**Участники:**

- Технологи/Инженеры (далее пользователь).
- Система (регистрация пользователя).

**Предусловие:** Система работает штатно, пользователь открыл панель HMI и выбрал раздел “Регистрация”. 


**Основной сценарий:**

1. Пользователь заполняет регистрационную форму, вводя личные данные: ID, имя, пароль, подтвердить пароль.
2. Пользователь подтверждает введенные данные нажатием на кнопку “Подтвердить регистрацию”.
3. Система проверяет ID пользователя.
4. Если ID есть в базе данных, система создает учетную запись пользователя.
5. Система выдает права пользователю на функционал в зависимости от типа пользователя (технолог/инженер).

   

**Постусловие:** Пациент успешно зарегистрировался и может войти в личный кабинет.


## **Use Case 2**

**Сценарий:** Принудительная остановка системы пользователем.

**Цель:** Зафиксировать id пользователя остановившего систему и причину остановки.

**Участники:** 

-  Технологи/Инженеры
- Система (система)

**Предусловие:** система работает в штатном режима, пользователь ввел свои данные и остановил работу системы.

**Описание:**

- Пользователь открывает HMI, вводит имя и пароль.
- Система проверяет корректность данных пользователя, выдает права на управление системы в ручном режиме.
- Пользователь нажимает кнопку “Принудительная остановка системы”.
- Система останавливается.
- Система сохраняет информацию данные о прогрессе остановленного процесса.
- Система передает данные пользователя, причину, дату и время остановки в следующие компоненты системы: MES, SCADA, IoT.
- Система включает звуковую и цветовую индикацию в комплексе.

  
  **Постусловие:** система успешно остановлена с сохранением данных о текущем прогрессе остановленного процесса, включила индикацию в комплексе и передала данные пользователя, причину, дату и время остановки в следующие компоненты системы: MES, SCADA, IoT.

## **Use Case 3**


**Сценарий:** Аварийный режим работы системы


**Цель:** Остановка системы в случае аварийной ситуации.


**Участники:** 

- Система
- Все пользователи комплекса

**Предусловие:** в системе возникла ошибка или пользователь активировал кнопку экстренной остановки.

**Описание:**

1. Система выявила ошибку при обработке сигналов с датчиков и экстренно завершила работу.
2. Пользователь комплекса нажал аварийную кнопку и прекратил работу системы.
3. Экстренная остановка системы, передача состояния в SCADA, IoT, MES.
4. Система включает светозвуковую индикацию в комплексе.

   
   **Постусловие:** система прекратила работу.


## **Use Case 4**


**Сценарий**: Корректировка технологических процессов  


**Цель**: Изменение параметров продукта. 


**Участники**:

- Система
- Технолог

  

**Предусловие**: система работает в штатном режиме, требуется изменение параметров для создаваемого продукта.


**Описание**: 

1. Технолог ввел свои данные в систему, получил права к изменению параметров системы.
2. Технолог заносит параметры в систему.
3. Система сохраняет id пользователя, описание изменений.
4. Система обрабатывает данные, моделирует поведение с измененными параметрами.
5.  При отсутствии ошибок в моделировании - изменяет алгоритм в связи с новыми параметрами.


**Постусловие**: Система функционирует исправно с новыми параметрами.


## **Use Case 5**


**Сценарий**: Интегрирование нового оборудования инженером АСУ ТП.


**Цель**: Интегрирование нового оборудования инженером АСУ (PLC,Sensors)


**Участники**: Инженер АСУ


**Предусловие**: Система принудительно остановлена.


**Описание**: 

1. Система остановлена, предварительно сохранена точка прогресса выполнения процесса. 
2. Инженер внедряет новый датчик.
3. Датчик отображается в PLC и передает данные в SCADA.
4. SCADA инициализирует датчик и передает данные в IoT.
5. MES система заносит новую единицу оборудования в базу данных.


**Постусловие**: оборудование успешно внедрено, все данные передаются в рамках ИС ЗСК.


## **Use Case 6**


**Сценарий**: Просмотр результатов за квартальный отчёт.


**Цель**: Успешно просмотреть отчёт о простоях оборудования, ошибках, кпд системы в системе визуализации.


**Участники**: 

- Оператор комплекса.
- Система (система визуализации Apache Superset)


**Предусловие**: система работает в штатном режиме.

**Описание**:

1. Оператор вводит данные ЛК и входит в систему.
2. Система выдает права пользователю.
3. Оператор создает отчетность по нужному для него периоду.
4. Система предоставляет всю необходимую информацию на дашборде.


**Постусловие**: Система исправно обработала и визуализировала данные за выбранный оператором период.


## **Use Case 7**


**Сценарий**: Уведомление об Warning-ах системы


**Цель**:  Предупредить пользователей о возможных проблемах в комплексе.


**Участники**: 

- Технолог
- Оператор
- Инженер АСУ 


**Предусловие**: Система работает исправно, но с небольшими изменениями, которые могут привести к аварии.


**Описание**: 

1. В физическом компоненте системы (PLC, Sensors) возникает проблема программного или физического уровня.
2. Уведомление приходит на SCADA систему оператору и инженерам АСУ.
3. Инженеры реагируют на данное предупреждение и исправляют его не допуская аварийной ситуации.  

**Постусловие**: система работает в исправном режиме, предупреждение снято.

# 3.2 Нефункциональные требования



## 1. Доступность 

  

- Система должна функционировать круглосуточно, 7 дней в неделю. Время простоя не должно превышать 0.1% времени в год.
- Система должна иметь встроенные инструменты мониторинга, которые будут отслеживать доступность всех компонентов системы в реальном времени.
- Время восстановления после сбоя не должно превышать 15 минут.

  

## 2. Целостность 

 

- Все данные системы должны резервироваться на 2 серверах, не реже одного раза в сутки. Резервные копии должны храниться в защищённом месте и быть доступны для восстановления в случае сбоя системы. Это обеспечивает сохранность данных и возможность восстановления в случае потери или повреждения данных.
- Обеспечить синхронизацию данных в БД системы 
  1. При регистрации пользователя данные о нем должны появляться одновременно в БД каждой из компонентов системы.
  2. При обновлении имени и/или пароля пользователя информация должна быть синхронно обновлена во всех БД где есть данный пользователь.
- Логирование изменений и инцидентов: фиксация всех изменений с указанием времени, пользователя и описания, хранение логов не менее 7 месяцев.


## 3. Производительность

  

1. Измененные данные должны быть применены в течение 3 секунд после тестовой модуляции системы.
2.  Система должна поддерживать одновременную работу не менее 1000 активных датчиков без деградации производительности.

  

## 4. Надежность


- Время восстановления после сбоя не должно превышать 5 минут для основных компонентов системы.
- Система должна обеспечивать среднее время между отказами не менее 90 дней для критических компонентов.


## 5. Безопасность

  

- Все пользователи должны проходить аутентификацию при входе в систему. Доступ к различной функциональности системы должен регулироваться с помощью ролей и прав доступа.
